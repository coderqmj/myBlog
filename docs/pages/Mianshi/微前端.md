### 2.qiankun和iframe去实现有什么区别

```
1.qiankun是基于single-spa 的微前端，通过JS去加载子应用，在主应用的DOM上挂在子应用的DOM。
2.qiankun提供了沙箱机制，隔离JS，防止子应用的全局变量污染
3.也提供了样式隔离

1.iframe本身就是一个天然沙箱，完全隔离JS和CSS，防止子应用之间的影响
2.iframe子应用之间通信困难，性能有问题，SEO也有问题

选择：
1.如果子应用是React，Vue这些单页应用，使用qiankun，可以提效
2.子应用是JQuery这些，或者老代码，或者子应用有第三方的页面，使用iframe有更好的隔离性
```

### 4.qiankunJS隔离

```js
1.主要是分为3种：snapshotSandBox（版本低）、legacySandBox（单个）、proxySandBox（多个）
2.防止子应用对window造成污染

snapshotSandBox：
	1.每次激活进去的时候会把windows的属性都存下来，然后离开页面的时候对比原先的快照，看哪些属性有差别
  2.对比到有差别就是修改过的，然后把修改过的存起来，然后window上面的数据还原成第一次进来记录的数据
  3.再次激活的时候，通过modifyProps属性可以恢复之前的window属性
  4.缺点：浪费性能，每次激活或者离开页面都要遍历window
  5.优化：每次只把修改后的存起来就行了
  
legacySandBox沙箱：
	1.使用proxy实现，兼容性有点问题
  2.两个应用一起运行window就乱了，相当于两个应用同时操作一个window，会发生冲突
  原理：
  	1.每一次操作window的时候，都会把修改信息和原来的状态给收集起来
    2.然后inactive的时候，就用收集到原来状态的map将window恢复
    3.再次激活的时候然后又恢复状态
  
proxySandBox沙箱：
	1.完全不操作window，给每个子应用创建沙箱
  2.只到自己的沙箱里面取用属性，且只有激活的时候才执行

class ProxySandBox{
    proxyWindow;
    isRunning = false;
    active(){
        this.isRunning = true;
    }
    inactive(){
        this.isRunning = false;
    }
    constructor(){
        const fakeWindow = Object.create(null);
        this.proxyWindow = new Proxy(fakeWindow,{
            set:(target, prop, value, receiver)=>{
                if(this.isRunning){
                    target[prop] = value;
                }
            },
            get:(target, prop, receiver)=>{
                return  prop in target ? target[prop] : window[prop];
            }
        });
    }
}
// 验证：
let proxySandBox1 = new ProxySandBox();
let proxySandBox2 = new ProxySandBox();
proxySandBox1.active();
proxySandBox2.active();
proxySandBox1.proxyWindow.city = 'Beijing';
proxySandBox2.proxyWindow.city = 'Shanghai';
console.log('active:proxySandBox1:window.city:', proxySandBox1.proxyWindow.city);
console.log('active:proxySandBox2:window.city:', proxySandBox2.proxyWindow.city);
console.log('window:window.city:', window.city);
proxySandBox1.inactive();
proxySandBox2.inactive();
console.log('inactive:proxySandBox1:window.city:', proxySandBox1.proxyWindow.city);
console.log('inactive:proxySandBox2:window.city:', proxySandBox2.proxyWindow.city);
console.log('window:window.city:', window.city);
// 输出：
// active:proxySandBox1:window.city: Beijing
// active:proxySandBox2:window.city: Shanghai
// window:window.city: undefined
// inactive:proxySandBox1:window.city: Beijing
// inactive:proxySandBox2:window.city: Shanghai
// window:window.city: undefined


```


### 7.预加载的原理

```
1.首先判断预加载是否开启，默认是开启的
2.也可以传一个数组，指定哪些是预加载的，也可以传一个参数，指定criticalApp预先加载
3.如果是true的话，会调用prefetchAfterFirstMounted(apps)，需要等第一个应用加载完之后去加载其他应用
4.就是监听single里面的 'first-mounted'事件，里面把没有加载的APP拿出来，进行加载
5.然后就是通过requestIdleCallback在浏览器空闲的时候再去执行额外的样式表和JS
```

### 6.single-spa的原理，优缺点

```
1.它是基于路由的加载的
2.当不同路由被激活时，会去加载我们对应的子应用，然后调用子应用的生命周期

缺点：
1、注册起来麻烦
2、没有预加载
```

### 7.qiankun优缺点

```
1.样式隔离
2.JS沙箱
3.预加载
4.大量开发者验证
```
